### <a name="_qmphm5d6rvi3"></a>**Решение**
Сценарий реализуется комбинацией паттернов CQRS + API Composition.

- Write-сервисы (Orders, Payments, Inventory, Logistics) продолжают публиковать доменные события в Kafka
- Добавлен Orders Query Service (read side), который потребляет события: OrderSubmitted, InventoryReserved/Failed, PaymentAuthorized/Failed, ShipmentScheduled/Failed, OrderCompleted/Cancelled, а также ProductUpdated/PriceUpdated для обогащения. Он формирует денормализованные проекции в OrdersView DB.
- BFF использует API Composition только по read-стороне: GET /me/orders и GET /me/orders/{id} обращаются к Orders Query Service; POST /reviews - к Reviews Service; POST /orders/reorder - к Orders (write).
- Для чеков Orders Query хранит 
eceiptUrl из Payments (или токен для запроса чека) и отдаёт BFF защищённую ссылку (pre-signed).
- Для отзывов добавлен Reviews Service (новый или существующий). Повторный заказ инициируется командой в Orders.

Основные компоненты и интеграции показаны на диаграмме. Read-модель спроектирована под быстрые выборки с пагинацией и фильтрами; проекции идемпотентны и переигрываемы из журнала событий (DLQ + replay).

### <a name="_bjrr7veeh80c"></a>**Альтернативы**
1) Только API Composition (без read-модели)
- Плюсы: минимальные изменения, нет дополнительных хранилищ.
- Минусы: высокая латентность (fan-out в Orders/Payments/Logistics/Catalog), хрупкость (каскад недоступностей), сложная кэш-стратегия, нет таймлайна статусов.

2) Полный Event Sourcing (источником истины являются события)
- Плюсы: полный аудит и воспроизводимость, гибкие проекции.
- Минусы: высокая сложность и стоимость миграции существующих сервисов; требует дисциплины схем и версионирования.

3) Репортинговая БД из write-реплики Orders
- Плюсы: просто стартовать.
- Минусы: сложно агрегировать статусы/чеки/доставку из других bounded context; денормализация смещается в runtime.

**Недостатки, ограничения, риски**
- Схемы событий: необходимы контрактные тесты и эволюция схем (avro/proto) с обратной совместимостью.
- GDPR/удаление персональных данных: проекции должны поддерживать удаление/анонимизацию по запросу.
- Нагрузка на Kafka/проектор: требуется мониторинг лагов и DLQ-процедуры.
